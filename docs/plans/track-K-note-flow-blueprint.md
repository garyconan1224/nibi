# 2. 内容处理

核心：把各种素材统一转成 **MD**。图中分支如下：

```
音频 ──→ 转码 ──→ MD

文字 ──→ MD

图片 ┬─ 文字型  ──→ 内容转写 MD
     └─ 非文字型 ──→ 视觉模型去做描述

视频 ┬─ 分离音频 ──→ 转码 ─────────────────→ MD ┐
     │                                          ├──→ MD
     └─ 画面截图 ──→ 图片内容简单识别 ──→ MD ──┘
        （每几帧截图设置页设置）
```

逐条对应原图文字：

| 素材 | 处理链 | 备注 |
|---|---|---|
| 音频 | 转码 → MD | — |
| 文字 | → MD | — |
| 图片·文字型 | 内容转写 MD | — |
| 图片·非文字型 | 视觉模型去做描述 | — |
| 视频·音轨 | 分离音频 → 转码 → MD | 与画面轨合并成最终 MD |
| 视频·画面 | 画面截图 → 图片内容简单识别 → MD | **每几帧截图设置页设置** |

> 视频两路（音频转写 + 画面识别）最终**合并为一份 MD**。

# 附录 C · 疑点④专项分析（图片 / 图文 是否分开）

> 用户要求：从「前期收集数据 → 最终结果页展示」整条链路思考。以下是分析与建议，**非定论，待用户拍板**。

## C.1 现状（已核对代码，2026-06-07）

| 阶段 | 现状 | 图片 vs 图文 是否已区分 |
|---|---|---|
| 采集（`kind_hint`） | `pipeline_tasks.py:1249` 有 `text / image_text / video / audio`；小红书、B站 opus → `image_text`，走专用适配器 | **已区分**（图文进来时是独立识别的） |
| 类型（`item.type`） | 只有 `image / text / video / audio` 四类（`note_assembler.py:44` 分支、`workspaces.py:450` 校验） | **未区分**：图文被并进了 `image`（无第 5 类 type） |
| 结果页 | 按 `item.type` 分流到 NoteShell | 跟着 type 走，**图文目前没有独立结果页** |

> 一句话：**图文在"进门"时分得清，但落到类型/结果页时被并进了图片。**

## C.2 数据本质区别（按用户原图的数据流）

- **图片**（§3.3）：输入是**纯图片**（一张/多张），文字是从图里「内容转写 / 视觉模型描述」**衍生**出来的 —— 图是主体、文字是产物。
- **图文**（§3.4）：输入是**原生图文混排**（小红书 / 公众号 / 网页文章），正文文字 + 配图都是原生的，要「富文本在合适位置加载图」**还原排版** —— 文字是主体、图穿插其中。

→ 展示形态确实不同：图片更像「图集 + 每图描述」，图文更像「带插图的文章」。

## C.3 三种处理方式（供选择）

- **方案 A：图文升格为独立第 5 类 type**（`image / text / video / audio / imagetext`）
  - 优点：概念最清晰，结果页彻底分开。
  - 成本：要动 type 枚举、`_ensure_valid_item_type`、采集端 map、路由、结果页分流等多处 —— **改动较大，且动数据结构**（按项目规则需你确认）。

- **方案 B（倾向推荐）：底层仍是 `image` type，加一个标记区分两种编排**（如 `is_gallery` / `source=xhs|opus` / 有无原生正文）
  - 优点：接得上采集端已有的 `image_text` 识别；**不动 type 枚举，改动小**；NoteShell 一个壳、按标记切换「图集编排」或「图文文章编排」两种渲染。
  - 缺点：图片/图文在"类型"上仍是同一类，靠标记区分。

- **方案 C：完全不分，一个「图片类」页面自适应**（有原生正文 → 图文文章排版；纯图 → 图集排版）
  - 优点：最省心，用户/系统无需关心是「图片」还是「图文」。
  - 缺点：自适应规则要可靠，边界情况（图里也有文字）可能判断不准。
